# Техническое задание

**Проект:** Сборщик отчётов с цветочных оптовых порталов и агрегация в Excel
**Кодовое имя:** `cvetopt`
**Версия документа:** 0.3 (черновик)
**Дата:** 2026-05-13
**Статус:** **Первая задача (v1):** только скачивание xlsx-отчётов по заказам с Biflorica — без разбора файлов и без `Auto_new.xls`. Этап агрегации — отдельно, после появления примеров.

> **Изменения в v0.3** — зафиксированы границы отбора `6 ≤ age ≤ 12`, вкладка «Все», идемпотентность через реестр скачанных `order_id`; первая итерация разработки ограничена только скачиванием.

---

## 1. Цели и контекст

### 1.1. Бизнес-цель
Автоматизировать сбор отчётов по сделкам со своих заказов на оптовых порталах. Сейчас оператор вручную отбирает заказы по дате вылета и скачивает xlsx по кнопке «Отчёт по сделкам». В перспективе данные из отчётов будут собираться в сводный Excel (`Auto_new.xls`) — это **не входит в первую задачу**.

### 1.2. Что делает система

**Первая задача (v1, в работе):**
1. Логинится на портал Biflorica (логин/пароль из `.env`).
2. Открывает страницу заказов, вкладка **«Все»**.
3. Отбирает заказы, у которых **дата вылета** даёт возраст **от 6 до 12 дней включительно** относительно «сегодня»: `6 ≤ (today − дата_вылета) ≤ 12`.
4. Для каждого отобранного заказа, **ещё не скачанного** (см. п. 6.4), скачивает xlsx «Отчёт по сделкам».
5. Складывает файлы в `data/downloads/biflorica/<YYYY-MM-DD>/` и **фиксирует ID заказа** в реестре успешных скачиваний.

**Этап 2 (позже):**
- Разбор содержимого скачанных xlsx и заполнение `Auto_new.xls` по образцу — после получения примеров файлов от заказчика. В коде и GUI на первом этапе aggregator не обязателен.

### 1.3. Что система НЕ делает в v1
- Не парсит и не изменяет содержимое скачанных xlsx.
- Не заполняет `Auto_new.xls`.
- Не делает закупку, не оформляет заказы, ничего не отправляет на портал.
- Не хранит историю цен/сделок в БД (только файлы на диске + реестр ID).
- Не скачивает фотографии.
- Не подключает del-mir и прочие порталы (отдельные задачи).

## 2. Глоссарий

| Термин | Значение |
|---|---|
| **Портал** | Внешний сайт-поставщик с авторизацией (логин/пароль) |
| **Заказ** | Запись в таблице «Заказы», имеет ID, даты, имя и т.д. |

| **Возраст (age)** | Целое число дней: `(today − дата_вылета)` в календарных днях |
| **Реестр скачанных** | Файл состояния со списком `order_id`, по которым отчёт уже успешно сохранён |
| **Прогон / job** | Один цикл: логин → список заказов → скачивание новых отчётов |
| **Auto_new.xls** | Будущий сводный файл (Этап 2), не в scope v1 |

## 6. Функциональные требования

### 6.1. Авторизация
Как в v0.2: `.env`, сохранение `data/sessions/biflorica.json`, повторный логин при протухшей сессии, поле Code не используется.

### 6.2. Отбор заказов (Biflorica)
1. Открыть страницу заказов.
2. Переключить вкладку на **«Все»**.
3. Установить диапазон дат на странице так, чтобы в выборку попали заказы с датой вылета в окне `[today − max_age, today − min_age]` (с запасом `list_buffer_days` по краям при необходимости). При `min_age=6`, `max_age=12` дата вылета лежит в `[today−12, today−6]`.
4. Обойти **всю пагинацию**.
5. Для каждой строки прочитать **ID** и **дату вылета**, вычислить `age`.
6. Оставить заказы, для которых **`min_age_days ≤ age ≤ max_age_days`** (по умолчанию **6 ≤ age ≤ 12**).

### 6.3. Скачивание отчётов
- Для каждого отобранного заказа, прошедшего проверку п. 6.4, инициировать скачивание («Отчёт по сделкам») и перехватить файл через `page.expect_download()`.
- Сохранить как `data/downloads/biflorica/<YYYY-MM-DD>/<order_id>__<flight_date>.xlsx` (имя можно уточнить по факту ответа сервера).
- Пауза 1–3 с между скачиваниями (рандом).
- Ошибка по одному заказу → лог, переход к следующему **без** записи в реестр.

### 6.4. Идемпотентность и реестр скачанных ID
**Решено заказчиком:** уже скачанные отчёты не качаем повторно; каждый успешно сохранённый заказ фиксируется по **ID заказа**.

Реализация:
- Файл реестра: `data/state/biflorica_downloaded.json` (или `data/state/downloads.json` с вложенностью по `portal_id`).
- Формат: JSON-массив или объект с множеством строковых ключей `order_id`; допускается хранить метаданные: `{ "order_id": "10720147", "downloaded_at": "2026-05-13T10:00:00Z", "path": "..." }` в массиве **или** только множество ID — на усмотрение разработки, минимум — **множество `order_id` для портала biflorica**.
- **Перед скачиванием:** если `order_id` уже есть в реестре → **пропуск** (не жать кнопку, не ждать).
- **После успешной записи файла на диск:** добавить `order_id` в реестр (атомарная запись: temp-файл + rename).
- Дополнительно: если файл с ожидаемым путём уже существует, но ID нет в реестре (ручное копирование / сбой) — **политика:** либо добавить ID в реестр без повторного скачивания, либо перескачать — в v1 предпочтительно **не перескачивать**, если файл существует и не нулевого размера; при расхождении логировать.

> **Сброс реестра:** вручную (редактирование JSON или кнопка «Очистить реестр biflorica» в настройках администратора) — опционально в v1, иначе правка файла на диске.

### 6.5. Антибот / устойчивость

## 10. Сценарии использования

### Сценарий 1. Первый прогон
1. Реестр пуст → все подходящие по возрасту заказы качаются.
2. После каждого успешного файла — запись `order_id` в реестр.

### Сценарий 2. Повторный прогон
1. Заказы с `order_id` из реестра — пропуск.
2. Новые заказы, попавшие в диапазон 6–12 дней — скачиваются.

### Сценарий 3. Протухшая сессия
Автоперелогин, затем как в сценарии 1 или 2.

## 13. Критерии приёмки v1

1. По кнопке в GUI выполняется прогон Biflorica: логин (или сессия), вкладка «Все», корректный отбор заказов с **`6 ≤ age ≤ 12`** по дате вылета.
2. Для каждого подходящего заказа с **новым** `order_id` скачивается xlsx в `data/downloads/biflorica/<дата>/`.
3. Повторный прогон **не** скачивает повторно заказы, чьи `order_id` уже в реестре.
4. Лог прогона виден в UI; ошибки по отдельным заказам не роняют весь прогон.
5. README: как задать `.env`, как запустить, где лежат файлы и реестр.
